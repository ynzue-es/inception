# Developer Documentation — Inception

## Setting Up the Environment from Scratch

### Prerequisites

Inside the virtual machine, install:

- **Docker Engine** (not Docker Desktop)
- **Docker Compose** (v2, integrated as `docker compose`)
- **make**
- **openssl** (for generating the TLS certificate if not already present)

On Debian:

```bash
sudo apt update && sudo apt install -y docker.io docker-compose-plugin make openssl
sudo usermod -aG docker $USER
# Log out and back in for the group change to take effect
```

### Repository Structure

```
.
├── Makefile
└── srcs/
    ├── .env
    ├── docker-compose.yml
    └── requirements/
        ├── mariadb/
        │   ├── Dockerfile
        │   ├── conf/
        │   └── tools/
        ├── nginx/
        │   ├── Dockerfile
        │   ├── conf/
        │   └── tools/
        ├── wordpress/
        │   ├── Dockerfile
        │   ├── conf/
        │   └── tools/
        └── tools/
            └── secrets/
```

Each service has its own directory containing a `Dockerfile`, a `conf/` folder for configuration files, and a `tools/` folder for entrypoint scripts.

### Configuration Files

**`srcs/.env`** — environment variables loaded by Docker Compose:

```env
DOMAIN_NAME=<login>.42.fr
DB_NAME=wordpress
DB_USER=wpuser
DB_HOST=mariadb
WP_TITLE=Inception
WP_ADMIN_USER=admin
WP_ADMIN_EMAIL=admin@example.com
WP_USER=editor
WP_USER_EMAIL=editor@example.com
```

**Secret files** — one password per file, no trailing newline:

```
srcs/requirements/tools/secrets/db_root_password.txt
srcs/requirements/tools/secrets/db_password.txt
srcs/requirements/tools/secrets/wp_admin_password.txt
srcs/requirements/tools/secrets/wp_user_password.txt
```

Create them manually. Example:

```bash
mkdir -p srcs/requirements/tools/secrets
echo -n 'StrongRootPass123' > srcs/requirements/tools/secrets/db_root_password.txt
echo -n 'StrongDbPass456'   > srcs/requirements/tools/secrets/db_password.txt
echo -n 'StrongWpAdmin789'  > srcs/requirements/tools/secrets/wp_admin_password.txt
echo -n 'StrongWpUser012'   > srcs/requirements/tools/secrets/wp_user_password.txt
```

**TLS certificate** — generated by the NGINX entrypoint or manually:

```bash
openssl req -x509 -nodes -days 365 \
  -newkey rsa:2048 \
  -keyout srcs/requirements/nginx/tools/nginx.key \
  -out srcs/requirements/nginx/tools/nginx.crt \
  -subj "/CN=<login>.42.fr"
```

## Building and Launching

### Makefile Targets

| Target | Description |
|---|---|
| `make` / `make all` | Build images and start containers in detached mode |
| `make down` | Stop and remove containers and networks |
| `make clean` | `down` + remove volumes and images |
| `make re` | `clean` then `all` |

Under the hood, the Makefile wraps `docker compose` commands pointing to `srcs/docker-compose.yml`. It also creates the host data directories if they don't exist:

```bash
sudo mkdir -p /home/<login>/data/wordpress
sudo mkdir -p /home/<login>/data/mariadb
```

### Build Details

All images are built from scratch using `docker compose build`. No image is pulled from Docker Hub except the base OS layer (`alpine:3.x` or `debian:bookworm`). The build context for each service is its own directory under `srcs/requirements/`.

## Managing Containers and Volumes

### Useful Commands

```bash
# Status of running containers
docker compose -f srcs/docker-compose.yml ps

# Rebuild a single service without cache
docker compose -f srcs/docker-compose.yml build --no-cache wordpress

# Open a shell inside a container
docker exec -it nginx sh
docker exec -it wordpress bash
docker exec -it mariadb bash

# View real-time logs for one service
docker compose -f srcs/docker-compose.yml logs -f mariadb

# List volumes
docker volume ls

# Inspect a volume
docker volume inspect srcs_wordpress_data
docker volume inspect srcs_mariadb_data

# Manually remove all project volumes (destructive)
docker volume rm srcs_wordpress_data srcs_mariadb_data
```

### Network

All three containers are connected to a single user-defined bridge network (named `inception` or similar in `docker-compose.yml`). This provides:

- Automatic DNS resolution between containers by service name (`wordpress` can reach `mariadb` by hostname).
- Isolation from the host and other Docker networks.
- Only NGINX exposes port 443 to the VM's network.

To inspect:

```bash
docker network ls
docker network inspect srcs_inception
```

## Data Persistence

### Where Data Lives

| Data | Docker Volume | Host Bind Path |
|---|---|---|
| MariaDB database files | `srcs_mariadb_data` | `/home/<login>/data/mariadb` |
| WordPress site files | `srcs_wordpress_data` | `/home/<login>/data/wordpress` |

Both volumes are defined in `docker-compose.yml` with a `driver_opts` local bind to the host paths above. This means:

- `make down` + `make` preserves all data.
- `make clean` destroys the volumes and all data with them.
- Backing up is as simple as copying `/home/<login>/data/` somewhere safe.

### Resetting the Database

To start fresh with a clean WordPress install:

```bash
make clean
make
```

This removes volumes, rebuilds images, and the MariaDB/WordPress entrypoint scripts re-initialize the database and site from scratch using the secrets and `.env` values.
